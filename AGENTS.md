# AGENTS.md

## Global Rules

### Design First
- 对于中大型功能改动，先补充或更新设计文档，再进入具体实现。
- 设计文档确认前，不直接开始大范围编码、联调或重构。
- 如果改动会影响后端接口、前端交互、数据模型、权限规则、部署方式或测试策略，必须先写设计文档。
- 只有用户明确要求跳过设计文档，或者改动足够小且不影响设计边界时，才可以直接实现。

### Design Document Minimum Checklist
- 背景与现状
- 目标与非目标
- 影响范围
- 数据模型或权限模型
- 后端实现方案
- 前端影响
- 测试与回归方案
- 风险与分阶段落地建议

### Delivery Sequence
- 先写设计文档。
- 再补或更新测试方案。
- 最后进行实现、联调和发布。

### Chinese Markdown Policy
- 仓库内的中文 Markdown 与文本文件统一使用 UTF-8 编码。
- 如果当前终端或脚本路径可能把中文写坏成 `?`，先做 canary 写入验证，再批量改文档。
- 中文文档优先整文件重写，不建议在不安全路径里逐行 patch 中文内容。
- 每批中文文档改动后，都要做乱码校验，至少检查是否出现异常的 `?` 或 `�`。
- 如果发现源文件已经被 `?` 污染，不要继续在坏文件上增量修改，应先从健康版本恢复后再编辑。
