# 文档级资源权限设计方案

## 1. 背景与现状

当前项目已经具备两层权限能力：
- 登录认证：基于 Spring Security + JWT
- 角色级鉴权：`admin` 可访问管理写接口，普通用户只能访问受限读接口范围

当前项目也已经具备权限策略存储能力：
- 通过 `/api/permissions/bind` 保存 `kb_permission_policy`
- 支持 `resourceType / resourceId / subjectType / subjectValue / effect`

但这些策略还没有真正作用到业务读链路。当前仍存在以下问题：
- 无论是否绑定了文档策略，只要用户完成登录，就可以读取文档详情
- 文档 chunks 不会按权限过滤
- 搜索结果不会按权限过滤
- QA 回答和 citations 也不会按权限过滤

因此，当前权限体系属于“策略已存储，但尚未生效”。

## 2. 目标

第一阶段实现“文档级资源权限”生效，覆盖以下读链路：
- 文档详情读取
- 文档 chunks 读取
- 搜索结果过滤
- QA 引用与检索结果过滤

实现后，系统要满足：
- 非管理员用户只能读取自己有权限的文档
- 搜索与 QA 不再泄露无权限文档内容
- 权限策略对读链路统一生效，而不是只在前端隐藏按钮

## 3. 非目标

第一阶段明确不做以下内容：
- chunk 级单独授权
- query log 级资源权限
- task 级资源权限
- 前端复杂菜单级资源控制
- 多租户权限体系
- 组织树、群组树、岗位树等复杂主体模型

## 4. 资源与主体模型

### 4.1 资源模型

第一阶段只支持一种资源类型：
- `document`

文档级权限将向下作用到以下数据：
- document detail
- document chunks
- search result item
- QA citations
- QA answer 的检索上下文

即：chunk 本身不单独配置权限，而是继承所属文档权限。

### 4.2 主体模型

第一阶段支持三类主体：
- `user`
- `role`
- `department`

其中：
- `user` 对应登录用户名或用户唯一标识
- `role` 对应当前已有的 `admin`、`viewer` 等角色
- `department` 对应用户所属部门

如果当前登录态暂时没有稳定携带 `department`，第一阶段可以先让 `department` 能存储，但只在用户上下文可解析后真正启用。

## 5. 判定规则

文档级资源权限采用以下判定顺序：

1. `admin` 默认放行
2. 读取目标文档对应的全部策略
3. 解析当前用户的主体集合，例如：
   - `user:tester`
   - `role:viewer`
   - `department:CreditReview`
4. 命中 `deny` 时，优先拒绝
5. 没有命中 `deny`，但命中 `allow` 时，允许访问
6. 完全没有命中时，默认拒绝

这意味着系统采用：
- `deny` 优先
- 白名单放行
- 默认拒绝

## 6. 后端设计

### 6.1 新增权限主体解析层

新增一个权限主体解析服务，例如：
- `PermissionSubjectResolver`

职责：
- 从当前认证上下文提取用户信息
- 生成可参与权限匹配的主体集合

输出示例：
- `user:tester`
- `role:admin`
- `department:CreditReview`

### 6.2 新增资源访问判定层

新增一个文档级权限服务，例如：
- `DocumentAccessService`

建议提供两个核心能力：
- `boolean canAccessDocument(UUID documentId, CurrentUser user)`
- `Set<UUID> filterAccessibleDocumentIds(Set<UUID> documentIds, CurrentUser user)`

职责：
- 拉取目标文档的权限策略
- 基于主体集合进行命中计算
- 返回单文档判定结果或可访问文档集合

### 6.3 接入文档详情链路

在文档详情查询前执行文档级权限校验：
- `/api/documents/{id}`

建议行为：
- 无权限时返回统一业务异常
- 可根据现有接口风格选择 `403` 或“伪装成不存在”

第一阶段建议优先使用 `403`，因为更容易调试与联调。

### 6.4 接入 chunks 链路

在 chunks 查询前执行文档级权限校验：
- `/api/documents/{id}/chunks`

行为与文档详情保持一致。

### 6.5 接入搜索链路

搜索的实现方式建议为：
1. 先执行正常检索
2. 收集结果中的 `documentId`
3. 过滤无权限文档对应的结果项
4. 重新计算最终 `total/items`

这样改动范围更可控，也适合当前项目结构。

### 6.6 接入 QA 链路

QA 链路建议复用搜索过滤后的结果：
- 无权限 chunk 不进入回答上下文
- citations 只保留有权限的引用
- 若过滤后无结果，则返回“未检索到足够证据”或等价提示

核心目标是：
- 不泄露无权限文档片段
- 不暴露无权限 citations

## 7. 前端影响

前端仍然只做角色级 UI 控制，不承担资源级授权判定。

前端需要配合的点主要是体验层：
- 文档详情无权限时展示稳定错误提示
- chunks 无权限时展示稳定错误提示
- 搜索和 QA 在结果被权限过滤后，仍保持可解释的空态或失败提示
- 权限绑定页继续由 `admin` 使用，不需要增加复杂资源菜单逻辑

因此，资源级权限的核心实现仍然在后端，前端只负责消费结果和展示提示。

## 8. 数据结构与接口影响

### 8.1 数据表

第一阶段不新增权限表，直接复用：
- `kb_permission_policy`

现有字段已经足够支撑：
- `resource_type`
- `resource_id`
- `subject_type`
- `subject_value`
- `effect`

### 8.2 权限绑定接口

第一阶段继续复用现有接口：
- `/api/permissions/bind`

后续如果需要更好的治理体验，可以再补：
- 查询某个资源当前策略列表
- 删除单条策略
- 分页查询策略

但这些不是第一阶段阻塞项。

## 9. 测试方案

### 9.1 后端集成测试

至少新增以下测试：
- `admin` 可读取受策略保护的文档
- 有 `allow` 的普通用户可读取文档
- 无匹配策略的普通用户不可读取文档
- 命中 `deny` 时优先拒绝
- 搜索结果会过滤无权限文档
- QA citations 会过滤无权限文档

### 9.2 前端回归

前端 Playwright 建议新增：
- viewer 打开无权限文档详情时的提示
- 搜索不再返回无权限文档
- QA 页面不再展示无权限引用

## 10. 分阶段落地建议

推荐按以下顺序实现：

### 阶段一
- 文档详情权限校验
- chunks 权限校验

### 阶段二
- 搜索结果过滤

### 阶段三
- QA 引用与检索结果过滤
- Playwright 回归补齐

这样可以把风险拆开，也方便逐层验证。

## 11. 风险与注意事项

需要重点注意以下风险：
- `department` 若未稳定进入认证上下文，第一阶段不要强依赖
- 搜索和 QA 的过滤必须发生在结果返回前，不能只在前端隐藏
- 若未来引入 chunk 级授权，需要重新评估当前“文档权限向下继承”的假设
- 若后续希望对外隐藏资源存在性，再考虑把部分 `403` 调整为“资源不存在”风格响应

## 12. 当前建议

当前最适合的落地目标是：
- 先实现文档级资源权限
- 只覆盖 `document detail / chunks / search / QA`
- 不扩大到 task、query log、chunk 级单独授权

这样可以用最小范围把权限体系真正从“可配置”推进到“可生效”。
