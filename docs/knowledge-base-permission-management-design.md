# 权限策略治理页设计方案

## 1. 背景

当前项目已经具备两类与权限相关的能力：
- 登录认证与角色级鉴权
- 文档级资源权限生效

同时，系统也已经具备策略写入入口：
- `POST /api/permissions/bind`

但是当前治理能力仍然比较粗糙：
- 前端只能整批覆盖写入策略，不能先查看当前已绑定的策略
- 不能删除单条策略，只能整批重绑
- 管理员很难判断某个文档当前到底生效了哪些 `allow/deny` 规则
- Playwright 与联调流程也缺少一条“查看 -> 删除单条策略 -> 再确认”的回归路径

这会带来两个问题：
- 实际治理成本偏高，容易误删或误覆盖已有策略
- 后续资源级权限继续扩展时，缺少稳定的运维与联调入口

## 2. 目标

本次仅补齐“权限策略治理”的第一层能力：
- 支持按资源查看当前策略列表
- 支持删除单条策略
- 保留现有整批绑定接口，作为批量覆盖入口
- 前端权限页升级成“查询 + 绑定 + 单条删除”的治理页

## 3. 非目标

本次明确不做以下内容：
- 不改动文档级权限判定规则
- 不引入策略分页
- 不引入策略编辑功能
- 不做 chunk / task / query log 级策略治理
- 不做审计日志或操作留痕

## 4. 用户场景

### 4.1 查看某个文档当前策略

管理员输入：
- `resourceType=document`
- `resourceId=<documentId>`

系统返回该文档当前全部策略，用于核对：
- 哪些主体被 `allow`
- 哪些主体被 `deny`
- 是否存在冲突策略

### 4.2 删除单条策略

管理员在列表中看到某条策略配置错误，例如：
- `role:viewer allow`

可直接点击删除，仅移除该策略，而不是整批覆盖所有策略。

### 4.3 整批覆盖绑定

管理员仍可继续使用现有整批绑定表单，一次性重写某资源的全部策略。

## 5. 接口设计

### 5.1 查询策略列表

新增接口：
- `GET /api/permissions`

请求参数：
- `resourceType`
- `resourceId`

响应数据建议结构：
- `resource_type`
- `resource_id`
- `items`

其中 `items` 每项包含：
- `policy_id`
- `subject_type`
- `subject_value`
- `effect`
- `created_at`

行为约束：
- 仅 `admin` 可访问
- `resourceId` 非法时返回统一参数错误
- 资源不存在时不强制校验资源主表，只按策略表返回空列表

### 5.2 删除单条策略

新增接口：
- `DELETE /api/permissions/{policyId}`

响应数据建议结构：
- `policy_id`
- `status`

行为约束：
- 仅 `admin` 可访问
- `policyId` 非法时返回统一参数错误
- 策略不存在时返回统一业务错误

### 5.3 保留整批绑定接口

继续保留：
- `POST /api/permissions/bind`

用途保持不变：
- 对单个资源的策略做整批覆盖写入

## 6. 后端设计

### 6.1 Service 层扩展

在 `PermissionService` 中新增：
- 查询指定资源当前策略列表
- 删除单条策略

### 6.2 Repository 层扩展

在 `KbPermissionPolicyRepository` 中新增：
- 按 `resourceType + resourceId` 查询并按 `createdAt` 排序
- 按 `policyId` 查询

### 6.3 DTO 输出

新增一个策略列表项响应 DTO，避免继续直接返回匿名 `Map`。

建议字段：
- `policyId`
- `resourceType`
- `resourceId`
- `subjectType`
- `subjectValue`
- `effect`
- `createdAt`

### 6.4 错误处理

删除单条策略时：
- 如果找不到策略，返回统一业务异常
- 前端可据此展示稳定错误提示

## 7. 前端设计

### 7.1 页面结构

权限页调整为三段：
- 顶部资源定位表单
- 中间当前策略列表
- 底部整批绑定表单

### 7.2 当前策略列表

列表字段：
- Policy ID
- Subject type
- Subject value
- Effect
- Created at
- Actions

操作列提供：
- `Delete`

### 7.3 交互规则

- 点击 `Load policies` 后拉取当前资源策略
- 整批绑定成功后自动刷新列表
- 删除单条策略成功后自动刷新列表
- 查询为空时显示明确空态
- 查询失败、删除失败时显示页面内联错误提示

### 7.4 权限边界

前端仍然只做角色级 UI 控制：
- 非 `admin` 直接显示 `Permission denied`
- 实际接口权限仍以后端为准

## 8. 测试方案

### 8.1 后端集成测试

至少补以下场景：
- `admin` 可查询策略列表
- `admin` 可删除单条策略
- 删除后再次查询，列表数量减少
- 非 `admin` 删除策略被拒绝

### 8.2 前端 Playwright

至少补以下场景：
- 管理员加载某文档策略列表成功
- 管理员删除单条策略成功
- 删除后列表更新

## 9. 落地顺序

推荐顺序：
1. 后端接口与测试
2. 前端权限页改造
3. Playwright 回归补齐
4. 文档同步

## 10. 当前建议

本次改动应聚焦在“治理能力补齐”，不要扩展权限模型本身。

也就是说：
- 不改 `deny` 优先、默认拒绝等判定规则
- 只补“看得见、删得准、回归可验证”的策略治理闭环
